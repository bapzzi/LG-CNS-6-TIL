# <LG CNS 6기] 4일차 TIL — Git과 GitHub의 차이, 버전 관리와 staging·commit

> TL;DR: 매일 GitHub로 TIL을 올리면서도 헷갈렸던 "git과 github는 뭐가 다른가"를 정리한 날. git = 로컬 버전 관리(VCS), github = 그 로컬을 온라인에 올려 공유하는 곳. 마침 내가 맥북·데스크탑 두 기기로 Claude Code 작업을 git으로 이어가며 버전 관리에 애먹던 경험과 정확히 맞물려서 개념이 확 붙었음.

## 오늘의 학습 키워드
- 버전 관리 시스템(VCS), 분산형 관리 시스템, git의 지향점(빠른 수행)
- git vs github (로컬 vs 온라인 공유 / 목적 차이)
- CLI vs GUI(Sourcetree), git 최초 설정
- git 로컬 명령어: init · add(staging) · commit · log · branch · merge · checkout

## 공부한 내용 (내 언어로 정리)

### 핵심개념 1 — 버전 관리와 git이 무엇인가
버전 관리는 **파일의 변화를 시간에 따라 기록**하고, **특정 시점의 버전을 다시 찾아올 수 있게** 하는 것. 이걸 해주는 게 버전 관리 시스템(VCS)이고, git은 그중 **분산형** 방식의 한 종류로 **빠른 수행**에 초점을 둠.

포인트는 "혼자 하면 굳이 안 써도 되지만, **다 같이 하는 프로젝트일수록 버전 관리의 중요성이 커진다**"는 것. 여러 사람이 같은 파일을 만지니 누가 언제 뭘 바꿨는지, 어떤 시점으로 되돌릴지가 중요해짐.

git을 쓰는 방식은 두 가지:
- **CLI** — 터미널에 명령어를 직접 입력.
- **GUI** — 프로그램으로 조작. 대표적으로 **Sourcetree**는 커밋 이력을 **그래프 형태로 시각화**해서 보여줌.

**git 최초 설정**(내가 누구인지 등록 + 기본 브랜치 지정):
```
git config --global user.name "이름"
git config --global user.email "이메일"
git config --global init.defaultBranch main
```

### 핵심개념 2 — git ≠ github (이게 오늘의 하이라이트)
매일 GitHub로 TIL을 올리면서도 둘의 관계가 뭉뚱그려져 있었는데 이번에 갈렸음.

- **git = 로컬**. 내 컴퓨터 안에서 버전을 기록·관리. **목적은 버전 관리.**
- **github = 온라인**. 그 로컬 저장소를 인터넷에 올려 관리·**공유**하는 곳. **목적은 공유.**

그래서 명령어도 층이 갈림:
- **로컬(git)에서 일어나는 것**: `git init`(저장소 생성) → `git add`(변경을 **staging 영역**에 올림) → `git commit`(staging한 것을 **하나의 버전으로 기록**) → `git log`(이력 보기) → `git branch` / `git merge` / `git checkout`(가지 치고 합치고 옮겨 다니기). **이 전 과정은 전부 내 컴퓨터 로컬에서 일어남.**
- **github 쪽 개념**: `repository`(저장소), `fork`(남의 repo를 내 쪽으로 복제), `pull request(PR)`(내 변경을 원본에 반영해달라 요청), `issue`(과제·버그 트래킹), `actions`(개발·배포 자동화, CI/CD로 이해).

`add`→`commit`의 감을 강의에서 "**캡슐에 넣으면 이력 추적이 가능**"하다고 비유했는데, staging에 담은 변경을 commit이라는 캡슐로 봉인해 시점으로 남기는 그림으로 이해됨.

### 실습 — git 데모
`git init`으로 저장소를 만들고, 야구팀 예시로 **lions 팀·tigers 팀 yaml 파일**을 만들어 `git status`로 현재 상태(추적/스테이징 여부)를 확인. 마지막엔 만든 걸 **Sourcetree에 연결**해 그래프로 보며 마무리. (yaml은 전에도 봐서 형식은 친숙한데 아직 완전히 손에 익진 않음 — 별도로 정리 필요.)

## 트러블슈팅 (막힌 지점 · 해결 과정)

**1. "git과 github가 정확히 뭐가 다른가"라는 개념 혼란 → 로컬 vs 온라인으로 정리**
- 문제: 매일 GitHub로 TIL을 올리고 있어서 대충 역할은 아는데, 둘의 **정의·경계**를 물으면 답이 막혔음.
- 원인: 늘 붙여 써서 "git 명령 = github에 올리는 것"으로 한 덩어리로 인식하고 있었음.
- 해결: 강의로 갈림. **git(로컬 버전 관리) → github(그 로컬을 온라인에 올려 공유)**. `init`·`add`·`commit`·`log`는 전부 로컬 사건이고, `push`처럼 원격으로 보내야 비로소 github가 관여. 목적도 다름(버전 관리 vs 공유). 3일차에 IDE≠언어를 정리했던 것처럼, 오늘은 도구의 **경계**를 하나 더 그은 셈.

**2. 내 실제 경험이 강의 개념과 겹침 — 버전 관리를 왜 신경 써야 하는가**
- 문제: 나는 Claude Code 작업을 **맥북·데스크탑 두 기기**에서 이어가려고 git으로 프로젝트를 관리하는데, 버전 관리를 제대로 안 하면 **작업이 꼬이거나 과거 코드를 잊는** 일이 잦았음.
- 원인: 한쪽에서 커밋·push를 안 하고 다른 기기에서 손대면 두 갈래가 생겨(갈라짐) 나중에 합치기 어려워짐.
- 해결: 그래서 "앉으면 pull, 끝나면 push" 식으로 순서를 신경 써 왔는데, 이번 강의가 **협업일수록 버전 관리가 중요**하다고 강조하니 내 경험이 이론으로 뒷받침돼서 인상 깊었음. 두 기기 = 사실상 나와의 협업이라, 이미 겪은 문제였던 것.

**3. 맥에서 실습 확인이 잘 안 됨**
- 문제: 데모를 따라 하는데 맥 환경이라 강의(다른 환경)와 화면·경로가 달라 이력 확인이 매끄럽지 않았음.
- 해결(예정): 우선 개념 위주로 따라가고, 맥 기준 `git status`·`git log` 출력 형태와 Sourcetree 연결을 내 환경에서 다시 재현해 볼 것.
