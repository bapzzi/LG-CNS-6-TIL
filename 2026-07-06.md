# <LG CNS 6기] 1일차 TIL — 웹 서비스 구조와 WS/WAS

> TL;DR: 웹의 큰 그림(사용자 → FE → BE → DB)과 각 역할, 요청·응답 흐름, WS vs WAS를 처음부터 정리. 낯선 용어를 예전에 바이브코딩으로 만든 앱(Vercel·Supabase·GitHub)에 대입하니 붙었다 — 특히 "API=FE와 BE의 대화 규칙"이 내 앱에서 React↔Supabase 잇던 그 부분이었음.

## 오늘의 학습 키워드
- 프론트엔드/백엔드/데이터베이스(FE·BE·DB), 개발자 진로(FE·BE·DB·DevOps)
- 웹서버(WS) vs 웹애플리케이션서버(WAS), 정적 vs 동적
- ORM(JPA·MyBatis), DBMS/RDBMS/NoSQL
- API

## 공부한 내용 (내 언어로 정리)

### 1강
웹은 **프론트엔드(FE)·백엔드(BE)·데이터베이스(DB)**로 나뉨. FE는 사용자랑 상호작용하는 유일한 공간이고 **UI**로 소통함. 사용자가 **request**를 보내면 **response**를 받는 과정으로 돌아감. FE·BE 둘 다 다루면 풀스택, 개발과 운영을 같이 하면 **DevOps**. 그래서 개발자 진로가 크게 FE·BE·DB·DevOps로 갈림. **API는 BE와 FE가 대화하는 규칙**임.

### 2강
사용자는 브라우저로 도메인을 통해 요청을 보냄. 여기서 **웹서버(WS)**와 **웹애플리케이션서버(WAS)**가 나뉘는데, WS는 정적 리소스, WAS는 동적 리소스를 관리함. WS가 클라이언트 요청을 받아 정적 리소스를 바로 주거나, 아니면 WAS로 요청을 넘김. WAS와 DB 사이 연결·용어는 아래로 정리:

| 용어 | 뜻 |
|---|---|
| JDBC | WAS ↔ DB 연결 규약 |
| ORM | JDBC를 편하게 쓰게 해줌 (JPA=객체지향 중심 · MyBatis=SQL 중심) |
| DB / DBMS | DB=물리적 저장 공간 · DBMS=거기 얹힌 관리 소프트웨어 |
| DBMS 종류 | 오픈소스 RDBMS · 상용 RDBMS · NoSQL |

마지막에 클라이언트가 받아서 **HTML·CSS·JS**로 렌더링함.

## 트러블슈팅 (막힌 지점 · 해결 과정)
오늘은 실습 에러가 아니라 용어가 안 붙는 게 문제였음. 비전공자라 생소한 단어가 많았는데, 예전에 만든 앱에 대입하니까 정리가 됐음.

- **문제**: 전에 바이브코딩 할 때 Supabase로 백엔드 만들고 Vercel로 배포·프론트를 했던 것 같은데, 이게 오늘 배운 FE/BE/DB 구조랑 어떻게 맞는지, Git·Supabase·Vercel이 합쳐져서 돌아가는 건지 헷갈렸음.
- **정리**: 합쳐진 게 아니라 역할이 나뉜 거였음.
  - **Vercel = 프론트엔드.** React 화면 띄워주는 곳.
  - **Supabase = 백엔드 + DB.** 강의 용어로 치면 WAS 역할이랑 PostgreSQL(RDBMS)을 묶어서 빌려주는 거임. 그래서 백엔드를 구축했다는 말이 맞았음.
  - **Git(GitHub) = 버전 관리 + 배포 트리거.** 코드 push하면 Vercel이 그 레포 보고 알아서 빌드·배포함.
- **연결**: 흐름으로 쓰면 GitHub에 코드 올리고 → push하면 Vercel이 프론트 배포하고 → 그 화면이 API로 Supabase랑 대화하는 구조였음. 오늘 배운 "API는 프론트랑 백엔드가 대화하는 규칙"이 딱 내 앱에서 React랑 Supabase 잇던 그 부분이었음. 추상적이던 API가 내 경험에 붙으니까 이해됨.
