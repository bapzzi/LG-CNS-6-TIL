# <LG CNS 6기] 1일차 TIL — 웹 서비스 구조와 ws/was

## 오늘의 학습 키워드
- 프론트엔드 / 백엔드 / 데이터베이스, 개발자 진로(fe·be·db·devops)
- 웹서버(ws) vs 웹애플리케이션서버(was), 정적 vs 동적
- orm(jpa·mybatis), dbms / rdbms / nosql
- api

## 공부한 내용 (내 언어로 정리)

### 1강
웹은 프론트엔드, 백엔드, 데이터베이스로 나뉨. 프론트엔드는 사용자랑 상호작용하는 유일한 공간이고, ui를 통해 사용자와 소통함. 사용자가 request를 보내면 response를 받는 과정으로 돌아감. 프론트엔드를 fe, 백엔드를 be로 부르고, 둘 다 다루면 풀스택 개발자. 개발과 운영을 같이 하면 devops. 그래서 개발자 진로가 크게 프론트, 백엔드, 데이터베이스, 데브옵스로 갈림. api는 백엔드랑 프론트엔드가 대화하는 규칙임.

### 2강
사용자는 브라우저로 도메인을 통해 요청을 보냄. 여기서 웹서버(ws)랑 웹애플리케이션서버(was)가 나뉘는데, ws는 정적 리소스를 관리하고 was는 동적 리소스를 관리함. ws가 클라이언트 요청을 받아서 정적 리소스를 바로 주거나, 아니면 was로 요청을 넘김. was랑 db 사이는 jdbc로 연결되고, 이걸 편하게 도와주는 게 orm. orm 도구가 jpa(객체지향 중심)랑 mybatis(sql 중심)임. 물리적인 공간이 db고, 거기에 소프트웨어가 얹히면 dbms. 종류는 오픈소스 rdbms, 상용 rdbms, nosql db로 나뉨. 마지막에 클라이언트가 받아서 html·css·js로 렌더링함.

## 트러블슈팅 (막힌 부분)
오늘은 실습 에러가 아니라 용어가 안 붙는 게 문제였음. 비전공자라 생소한 단어가 많았는데, 예전에 만든 앱에 대입하니까 정리가 됐음.

전에 바이브코딩 할 때 supabase로 백엔드 만들고 vercel로 배포·프론트를 했던 것 같은데, 이게 오늘 배운 fe/be/db 구조랑 어떻게 맞는지, git·supabase·vercel이 합쳐져서 돌아가는 건지 헷갈렸음.

정리하니까 합쳐진 게 아니라 역할이 나뉜 거였음.
- vercel = 프론트엔드. react 화면 띄워주는 곳.
- supabase = 백엔드 + db. 강의 용어로 치면 was 역할이랑 postgresql(rdbms)을 묶어서 빌려주는 거임. 그래서 백엔드를 구축했다는 말이 맞았음.
- git(github) = 버전 관리 + 배포 트리거. 코드 push하면 vercel이 그 레포 보고 알아서 빌드·배포함.

흐름으로 쓰면 github에 코드 올리고 → push하면 vercel이 프론트 배포하고 → 그 화면이 api로 supabase랑 대화하는 구조였음. 오늘 배운 "api는 프론트랑 백엔드가 대화하는 규칙"이 딱 내 앱에서 react랑 supabase 잇던 그 부분이었음. 추상적이던 api가 내 경험에 붙으니까 이해됨.
